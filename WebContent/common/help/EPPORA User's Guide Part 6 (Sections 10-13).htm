10.3 Optimal Resource Allocation Process

In many organizations there is significant resources not applied to organizational activities or project tasks even though the employees are charging the organization for their time at their location of work. In every major organization EPSs staff has worked or consulted with 100s of organizations. We have noticed that if the typical staff member is being paid for forty hours a week to work that the staff member was actually working less than 70% of their normal working time. These missing hours were spent:

  Discussing news, movies, television shows, websites and sporting events with fellow workers

  Discussing personal (family and friends) issues and successes with fellow workers

  Waiting for fellow project team members to complete an assignment (dependencies) prior to enabling them to work

10.4 Measuring Job Performance Process

 

If all the project team members are building identical gizmos, than measuring performance would be simply tallying how many gizmos a project team member built on average per day. The individuals who created the greatest number of gizmos would be the most productive. Unfortunately there are very few, if any, projects that build identical gizmos. Too often project team members feel that their productivity does not have a direct correlation with their remuneration. Rather their relationship with their boss or their bosss boss along with their ability to tout their accomplishments as opposed to their actual accomplishments is major influences in determining salary adjustments and bonuses. It would be beneficial for nearly every organization conducting projects if there were an objective method for measuring an individuals productivity. This is exactly what EPPORA provides as described previously in Sections 5i through 5k. The major purpose of this measurement is to improve an organizations use of its precious labor resources, by assigning the most productive resources to the most important project tasks. A side benefit of this is that management can now look at scientific data when assessing salary adjustments and bonuses. Productivity isnt the only factor in determining how well someone should be compensated, but it is one of the key if not the key factor. An additional benefit is that if individuals believe their productivity is the major contributing factor to their financial success, they will tend to work hard and waste less time at work talking about non-work issues and events.

10.5 Estimating Project Costs

 

There are numerous ways different industries and organizations within an industry estimate project costs. In the IT industry it is often how many lines of code in the application times some cost per line of code. Another method involves the complexity of the application based on the depth of nesting. In construction it might be how many square feet times some quality factor times other factors. Consider the following two statements:

 

1.     What a project is all about is its requirements.

2.     The cost of implementing a project is the cost of implementing all the projects schedule tasks.

 

In EPPORA we allow the projects Business Analyst, Project Manager or Project Portfolio Manager to link each low-level requirement to the schedule tasks needed to accomplish the requirement. Each task that is linked to a requirement contains a specification as to what percentage of the task is used for the requirement it is being linked to. Since each low level task has associated with it labor costs, inventory costs (if any) and other costs (if any), it is simple in EPPORA to calculate the cost of a project, the remaining cost of a project, the cost of a requirement (low-level or parent) or the remaining cost of a requirement.

12.     Scripting Processor

EPS uses IOPUSs iMacro product to help support its testing activities. iMacro is used by EPSs software engineers and test engineers to assure that the EPPORA product satisfies each and every requirement. The execution and analysis of the test cases for a product the size of EPPORA is a timely, error-prone, expensive activity. To complicate matters, nearly all software experts agree that when even a minor change is made to a software product, to verify the product prior to a release, one should re-run the entire test suite (to minimize regressions). The EPPORA system is a complex, large application. Since EPPORA is a tool providing critical prioritization, allocation of precious labor resources and costing data, it is imperative that the results EPPORA provides are valid and new releases can be verified in a timely manner.

The cost of implementing the initial test suite with iMacro was slightly more expensive than testing EPPORA by a set test engineers running the test procedures (estimated at 10-20% more). The payoff in creating tests using iMacro occurs when the tests are run more than once. The cost for subsequent testing is a small fraction (less than 1%) of the initial test development cost. In addition, with iMacro, EPPORA is able to test new releases in a small fraction of the time it would have taken using multiple test engineers. It is also easy to enhance the testing of EPPORA as new features are added or latent defects occur.

iMacro is used in its record mode to specify tests to be executed by EPPORA. iMacro simplifies the task of producing executable actions that are repeatable by the EPPORA system. iMacro is independent of the EPPORA platform or Internet browser. iMacro can emulate any EPPORA keystroke or mouse click a user can perform by processing a text file. The text file contains a series of scripting language commands that instruct EPPORA to perform various actions, such as selecting an object, pressing a radio button, entering characters into a text edit object, selecting a check box, etc. In addition, the text-scripting file can include references to other files within it that perform additional actions. iMacro can take a snapshot of the screen and create a stylized text file representation of the page. iMacro can also compare two text files and report any differences. iMacro keeps a log that lists all errors encountered, as well as other user supplied and script supplied information.

13.     Automated Activities

EPPORA performs numerous behind the scene activities even when there are no users actively using EPPORA. These activities take place at the end of the business day and usually take less than an hour of wall clock time. The activities include:

a. Each Project

                                                i.    Verify that a Project Manager is specified

                                               ii.    Verify that a Business Analysts is specified

                                              iii.    Verify that a Project Portfolio Manager is specified

                                              iv.    Verify that a Sponsor is specified

                                               v.    Verify that the requirements are specified

                                              vi.    Verify that the schedule is specified

b.     Each Projects Schedule

                                                i.    Verify that each task starts with a verb

                                               ii.    Verify that each low-level task has generic labor resources assigned

                                              iii.    Verify that for the current allocation period each low-level task has an assigned actual resource for each required labor category

                                              iv.    Verify that the number of fixed dates for low-level task is within the organizations threshold

                                               v.    Verify that each low-level task has no resource assigned for over the maximum effort per low-level task (default is 40 hours)

                                              vi.    Verify that no high-level tasks (parent task) has resources assigned

                                             vii.    Verify that the schedule has been updated within the Schedule Update Period

                                            viii.    Verify that no deliverable is late

                                              ix.    Verify that no deliverable is projected to be over a week late

                                               x.    Verify that no milestone is late

                                              xi.    Verify that no milestone is projected to be over a week late

                                             xii.    Verify which deliverables were completed today

                                            xiii.    Verify which milestones were completed today

  Each Projects Requirements

                                          i.    Verify that each requirements description contains the word shall

                                         ii.    Verify that each requirements description does not contain any adjectives

                                        iii.    Verify that each requirements description does not contain any adverbs

                                        iv.    Verify that each requirements description is not a compound statement

                                         v.    Verify that each requirements description does not exceed the maximum number of words allowed

                                        vi.    Verify that each low-level requirement is mapped properly to one or more low-level project tasks

  Each Projects Requirements Mapping

                                          i.    Verify that each low-level requirement is mapped to one or more low-level project tasks

                                         ii.    Verify that the percentage of each low-level task used relative to mapping is 100%

  Each User

                                          i.    Verify that each user has a Supervisor specified (except if the user has a job title of CEO or President)

                                         ii.    Verify that each user has one or more labor categories specified

                                        iii.    Verify that each user has a work telephone number specified

                                        iv.    Verify that each user has an email address specified

                                         v.    Verify that no user is overloaded for each day of the allocation period
